# 验证码登录改造方案（OTP Login）

> 目标：把"账号密码登录 + 下发密码短信"整体改为**手机号 + 短信验证码登录**，PC 后台与移动端都改，**完全弃用密码**。
> 本文件是实现规格，建议在**全新会话**中按此执行（先设计确认 → 实现 → 自测通过再交付）。

## 一、用户已确认的决策

1. **范围**：移动端 + PC 后台 **都改**为验证码登录。
2. **绑定手机号**：每个登录主体绑手机号。
   - `admin` 超管 → **13601667307**
   - 其他种子账号（operator/auditor/product/branch/boss 等）→ 用**假号**（如 138000000xx）
3. **硬编码验证码**：
   - **正式环境（NODE_ENV=production 且无 dev 标记）**：**禁止**任何硬编码，必须走真实短信。
   - **测试/本地开发**：增加一个**硬编码验证码 `8888`**，任何手机号输入 8888 即可登录（免发短信，方便调试）。
4. **完全弃用密码**：删除/停用账号密码登录链路，不再 `sendAccountPasswordSms`。

## 二、关键设计

### 2.1 验证码存储（Redis）
- `otp:code:<phone>` = 验证码，TTL **300s**（5 分钟）
- `otp:resend:<phone>` = 重发锁，TTL **60s**（60 秒内不可重复发）
- `otp:attempt:<phone>` = 校验失败次数，超过 **5 次**锁定该手机号 10 分钟，防爆破
- 校验成功后立即删除 `otp:code:<phone>`

### 2.2 硬编码验证码开关
- 新增环境变量 `OTP_DEV_BYPASS_CODE`（仅在**非 production**或显式 `OTP_DEV_BYPASS=true` 时生效）。
- 取值：测试/本地 `.env.development` 设 `OTP_DEV_BYPASS_CODE=8888`；`.env.production` **不设置**。
- 逻辑：`if (devBypassEnabled && input === OTP_DEV_BYPASS_CODE) 通过`，否则走 Redis 校验。
- 守卫：`devBypassEnabled = process.env.NODE_ENV !== 'production' || process.env.OTP_DEV_BYPASS === 'true'`，且 production 下即使误设也要再做一层拦截，确保线上无后门。

### 2.3 短信发送
- 用新申报的**验证码模板**（变量名必须是 `code`），新增 env：
  - `ALIYUN_SMS_LOGIN_TEMPLATE_CODE=SMS_xxxxxxxxx`（用户申报通过后填）
- 复用现有 `src/lib/sms.js` 的客户端构造，新增 `sendLoginCodeSms(phone, code)`，`templateParam = { code }`。
- 删除/废弃 `sendAccountPasswordSms` 的调用点（建员工时不再发密码）。

### 2.4 接口
- 后台（PC）：
  - `POST /api/auth/send-code` body `{ phone }` → 发码（含 60s 限流）
  - `POST /api/auth/sign-in` 改为 body `{ phone, code }` → 校验 → 签 JWT（保留现有 cookie/JWT 机制与 `COOKIE_INSECURE`）
- 移动端：
  - `POST /api/mobile/auth/send-code` body `{ phone }`
  - `POST /api/mobile/auth/sign-in` 改为 `{ phone, code }`
- 移除：`/api/auth/change-password` 等密码相关接口（或停用返回 410）。

### 2.5 数据 / 账号
- `admin_staff` 已有 `phone` 字段。改造点：
  - 登录主体从 `account` 改为 **`phone`**（手机号唯一索引）。
  - 种子数据 `seedAdminStaff`：admin 的 phone 设为 **13601667307**，其余用假号；可去掉 password 或留空。
  - 已存在的库：写一个幂等迁移把 admin 的 phone 更新为 13601667307。
- **确认移动端登录主体**：是 `admin_staff`（门店账号）还是 `members`？实现第一步要先查清 `/api/mobile/auth/*` 当前用的是哪张表。

### 2.6 前端
- 后台登录页 `src/app/auth/[[...sign-in]]` / `features/auth`：改为「手机号 + 验证码 + 发送验证码按钮（60s 倒计时）」。
- 移动端登录页同理。
- 移除密码输入相关 UI。

## 三、测试计划（实现后我必须自测通过）
1. `send-code`：调用返回成功；Redis 写入 `otp:code:<phone>`，TTL≈300s；60s 内重发被拒。
2. **dev 硬编码**：`.env.development` 下，任意手机号 + `8888` 能登录换到 JWT。
3. **prod 无后门**：production 配置下，`8888` 必须**失败**，只有真实验证码通过（用 Redis 实测）。
4. 真实短信：用申报通过的验证码模板发到真实号，`QuerySendDetails` 状态=3 成功。
5. 登录全链路：phone+code → JWT cookie（沿用 COOKIE_INSECURE 逻辑）→ 能访问 `/dashboard`。
6. 密码登录入口已失效（旧接口返回 410/404）。
7. 防爆破：连续错误 5 次后锁定。
